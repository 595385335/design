CEBus Device Access API

Status of this Memo

This document specifies the CEBus Device Service and the CEBus Raw Service for the Open Services Gateway Initiative, and requests discussion and suggestions for improvement. Distribution of this document is unlimited. This RFC answers RFP-0011.

Abstract

In order to allow OSGi framework users to interact with CEBus physical devices by using the Java language, a standard CEBus Device Access service is desirable. This RFC describes the API of the CEBusDevice and the API of the CEBusRawService. Those two services put together should meet the requirements of all CEBus device vendors.

Acknowledgment

Many thanks to Tommy Bohlin (Gatespace), BJ Hargrave (IBM), Peter Kriens (Ericsson) and Christian Kruzke (Sun) who have been -- and continue to be -- very helpful in the drafting of this API proposal.

Introduction

The CEBus Device Access API contains mainly two services. The first one -- CEBusDevice -- defines a conventional set of methods that all CEBus devices' logical representations will support. It will be used by the driver developpers to get access to the CEBus capabilities without having to deal with any low level transmission/reception details. The second one -- CEBusRawService -- offers low level mechanisms to be able to send and receive messages on a CEBus network. It targets an experienced audience that wants a raw access to the CEBus network.

Motivation and Rational

CEBus device developers need a standard way to interact with the OSGi framework.

Technical Discussion

The two services that are integrated in the CEBus Device Access Bundle provide the following capabilities and features:

 

·         Offers the capability of sending messages synchronously (with CEBusDevice) or asynchronously (with CEBusRawService).

·         Offers two levels of access to the users: a high level access via CEBusDevice and a low level access via CEBusRawDevice.

·         Offers a common set of methods to interact with a logical representation of a CEBus device.

·         Offers a reception mechanism that allows interested entities to receive all message results/rejects from the physical network (via the CEBusRawService).

·         Offers a common way of associating the message request and the message result/reject received from the network via the CEBus raw service event mechanism.

 

The API for the CEBus Device service and for the CEBus Raw service are described in the Java source files and Javadoc HTML which are included as part of this RFC.

Security Considerations

No security considerations are discussed in this document. The security considerations that are related to this API proposal depend on the unregistration of all dependant layers of Device. However, the framework specification already covers this point. Therefore, these mechanisms will be used. Note that the HPnP/CEBus defines means for secure transactions through encryption. In addition, note that no CEBus implementation currently supports authentification and encryption. In any case, such security considerations are the responsibility of the underlying stack and not the API.
