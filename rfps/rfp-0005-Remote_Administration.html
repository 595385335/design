Services Framework Remote Administration RFP

Status of this Memo

This document suggests the following extension to the OSGi specification for the Open Services Gateway Initiative, and requests discussion. Distribution of this document is unlimited.

Abstract

A key aspect of the Open Services Gateway Initiative Services Framework[1] is the ability to administrate the framework under the control of a remote managing system. This RFP provides some general goals and guidelines and solicits technical submissions that will lead to defining an administration capability and API for the Services Framework.

Acknowledgment

The author wants to acknowledge the contributions of the members of the Connected Alliance and OSGi. 

Introduction

The Remote Administration API enables a managing system to remotely manage and configure the bundles and services on a target device.

Motivation and Rational

The primary goal of the Remote Administration API is to remotely manage and configure bundles and services on a target device. The Remote Administration API should meet this goal as follows. First, it should provide a consistent application programming interface (API) to use during development of device management applications. Second, it should provide the flexibility for implementors to "plug-in" different device management protocols below the Remote Administration API.

Another goal of the Remote Administration API is to enable implementors to deploy implementations in a variety of devices; the different hardware characteristics of the devices affect the amount of code that can run on these devices. Therefore, the Remote Administration API should require a minimum of components on the managed device; however, an implementor has the flexibility to utilize additional components in devices that can accommodate these components.

Technical Discussion

Administration of the Services Framework must provide, at a minimum, the following capabilities and features:

  • Remotely administer the Services Framework
  • Remotely configure services on a managed device.
  • Minimize the software requirements for a managed device.
    • The framework should only require the Java profile[2] for the Services Framework.
    • Enable services to share common configuration properties.
  • Scalable for mass deployment and configuration.
    • Services must be remotely configurable without a user interface.
    • Services may be remotely configurable with a user interface.
    • Enables configuration of services when the Services Framework on the target device is down or unavailable.
    • Enables an administrator to configure properties en masse for groups of devices.
    • Does not require downloading of communication classes from every managed device.
  • Upward compatible.
    • De-couples the configuration state from services.
    • Does not require serialization because a serialized object might not be compatible with all current or future JVM's.
  • Protocol neutral.

Security Considerations

As a request for specification proposals, this document has no direct security considerations.  However, the resulting Administration capability and API should undergo a security review before approval of the API.
