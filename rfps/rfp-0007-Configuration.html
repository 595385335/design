Services Framework Configuration RFP

Status of this Memo

This document suggests the following extension to the OSGi specification for the Open Services Gateway Initiative, and requests discussion. Distribution of this document is unlimited.

Abstract

A key aspect of the Open Services Gateway Initiative Services Framework[1] is the ability to configure the services on the device. This RFP provides some general goals and guidelines and solicits technical submissions that will lead to defining a configuration capability and API that programs can use to get and set configuration properties.

Acknowledgment

The author wants to acknowledge the contributions of the members of the Connected Alliance and OSGi. 

Introduction

The Configuration API enables a program to get and set configuration properties on a device.

Motivation and Rational

The primary goal of the Configuration API is to enable services to get and set configuration properties. The Configuration API should meet this goal as follows. First, it should provide a consistent application programming interface (API) to use during development of programs. Second, it should provide the flexibility for implementors to "plug-in" different persistent storage mechanisms below the Configuration API.

A typical product will be composed of many programs and services. If a product is re-designed in a future version and split into a different set of programs and services, the configuration state should still be usable. Therefore, another goal of the Configuration API is to de-couple programs and services from configuration.

Technical Discussion

Configuration Services must provide, at a minimum, the following capabilities and features:

  • Enable services to get and set configuration properties.
  • Support notification of events for property changes (e.g. insertion, removal, update)
  • Support the Java String, int, long, byte[], byte, and boolean property types.
  • Minimize the software requirements for a managed device.
    • The API should only require the Java profile[2] for the Services Framework.
    • Enable services to share common configuration properties.
    • Does not require serialization
  • De-couples the configuration state, storage location, medium, format and policies from services.

Security Considerations

As a request for specification proposals, this document has no direct security considerations.  However, the resulting Configuration capability and API should undergo a security review before approval of the API.
