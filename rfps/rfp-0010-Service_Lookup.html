Service Lookup Service

Status of this Memo

This document suggests the following extension to the OSGi specification and requests discussion. Distribution of this document is unlimited. 

Abstract 

The ServiceSpace [1] API has been limited in functionality to allow even very small devices to utilize it. This document requests and RFC that will extend the ServiceSpace API with a facility to find services based on selection criteria. 

Introduction 

The ServiceSpace API does not support a lookup facility for services. Services can only be obtained via the ServiceDescription. However, in many cases there is a need for a facility where a service can be found via its property values instead of its name. 
For example, a print service could register its service with a location property. This would allow any other service that wants to print to present a list of printing services that are close to a certain location. 

This RFP seeks RFCs that propose a service that can supply a filter mechanism for services. 

Terminology

A filter is a construction that asserts one or more values on a property. 
A property is a named attribute of a service. Each attribute can contain 0 or more values. 
A value can be an instance of different data types. 

Technical Discussion

The API must have:
  1. An OSGi service where other services can be enumerated that match a filter.
The API should have:
  1. The API should fit in the standard OSGi environment as described in [1] and [2].
  2. A standard set of attributes plus a mechanism to extend this
  3. Well defined rules for matching different data types 
  4. Filters should support constructs like AND, OR and NOT.
  5. Filters should support sub expressions
  6. A service property must be allowed to have multiple values
  7. Magnitude comparisons (>,<) should be supported for the appropriate attributes.
The API could have:
  1. API's based on earlier standards are preferable.
  2. The minimum implementation must be possible to make it small (e.g. <8K).
  3. It must be possible to construct a filter in run time.
  4. Filters should be human readable
  5. A mapping to existing service lookup mechanisms: SLP [5], SSDP[6], JINI [4], LDAP [3].
  6. Regular expression matching, approximate matching, presence testing.

Security Considerations 

The list of services should only contain services to which this service has access to.
