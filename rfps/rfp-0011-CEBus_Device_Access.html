CEBus Device Access API RFP

Status of this Memo

This document suggests a Device Access API be written to allow OSGi bundle access to CEBus devices “connected” to the OSGi device and requests discussion. Distribution of this document is unlimited.

Abstract

In order to allow OSGi bundles to interact with CEBus physical devices by using the Java language, a standard CEBus Device Access API is desirable. This RFP provides some general goals and guidelines and solicits technical submissions that will lead to defining such an API.

Acknowledgment

Many thanks to Tommy Bohlin (Gatespace), BJ Hargrave (IBM), Peter Kriens (Ericsson) and Christian Kurzke (Sun) who have been -- and continue to be – very helpful.

Introduction

The CEBus Device Access API should fit within the OSGi Device Access architecture and provide the standard low level device definition for CEBus devices as well as any additional API necessary to interact with the CEBus network.

Motivation and Rational

OSGi bundle developers looking to author bundles which refine or interact with CEBus devices need a standard way to interact with those devices.  

Technical Discussion

The CEBus Device Access API should provide the following capabilities and features: 

·         Define the CEBusDevice interface which extends the Device interface from the Device Access architecture.

·         Offers the capability of sending messages to CEBus devices.

·         Offers two levels of transmission (one with limited customizable fields and one with extended customizable fields).

·         Offers the capability of receiving message results/rejects.

·         Offers two levels of reception of message results/rejects: global reception and source addressed reception.

·         Offers a reception mechanism that allows interested device logical representations to receive message results/rejects.

·         Offers a common set of methods to interact with a logical representation of a CEBus device.

·         Offers a common way of associating the message request and the message result/reject received in response.

Security Considerations

As a request for specification proposals, this document has no direct security considerations.  However, the resulting API may undergo a security review before approval of the API.

 

Note that the HPnP/CEBus defines means for secure transactions through encryption. In addition, note that no CEBus implementation implementing the full CEBus stack with encryption currently exists. In any case, such security considerations are the responsibility of the underlying stack and not the API.
